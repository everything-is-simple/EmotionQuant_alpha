# EmotionQuant 核心原则（Spiral 版）

**版本**: v5.2.0  
**最后更新**: 2026-02-28  
**状态**: 单开发者执行基线

---

## 1. 项目定位

- 单开发者、个人使用优先
- 工程目标：每圈交付可运行闭环，不追求“文档完美”
- 管理目标：低维护成本、可持续推进

---

## 2. 核心原则

1. 情绪优先，单指标不得独立决策。
2. 本地数据优先，远端只补采。
3. 路径与密钥必须配置注入。
4. 严守 A 股规则（T+1、涨跌停（主板10%/创业板科创板20%/ST 5%）、交易时段、申万行业）。
5. 闭环优先于扩张（run/test/artifact/review/sync）。
6. 文档最小化，单一事实源优先。

### 2.1 技术指标边界（与系统铁律一致）

- 禁止：单一技术指标直接触发交易决策。
- 允许：技术指标仅用于对照实验或辅助特征，且必须与情绪因子联合验证。
- 说明：历史 archive 中“零技术指标绝对禁令”不属于现行治理口径。

---

## 3. 治理强度分级

### 3.1 默认（Spiral Lite）

- 流程：`Scope -> Build -> Verify -> Sync`
- 文档最小集：`requirements.md`, `review.md`, `final.md`

### 3.2 升级（Strict 6A）

仅在高风险变更启用：

1. 交易执行路径重大改动
2. 风控规则重大改动
3. 数据契约破坏性变更
4. 关键外部依赖影响主流程

执行权威入口：`Governance/steering/6A-WORKFLOW.md`

---

## 4. 质量门控

- 必须门：命令可运行、测试可复现、产物可检查
- 合规门：硬编码检查、A 股规则检查、本地数据检查
- 原则：有效测试优先于覆盖率数字

---

## 5. 同步约束（最小集）

每阶段强制同步：

1. `Governance/record/development-status.md`
2. `Governance/record/debts.md`
3. `docs/roadmap.md` 对应阶段状态
4. `docs/cards/` 对应卡片勾选

设计文档仅在实现与设计冲突时同步修正。
